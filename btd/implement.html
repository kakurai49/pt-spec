GitHub Issue＝Codexへのインプット（プロンプト）設計

以下は、GitHub Issue＝Codexへのインプット（プロンプト）としてそのまま運用できる形に、CIBG v0.1 実装を Codexが実装可能な粒度まで分解し、さらに Issue（＝Codex入力）設計をまとめたものです。

Codex向けの原則

  • 検証できる手順（pytest等）を必ず書く
  • 大きい作業は小さいIssueに分ける（レビューもしやすい）

これはCodex公式ドキュメントでも推奨されています。 ([OpenAI Developers][1])
また、Codexはリポジトリ内の AGENTS.md を事前に読み、そこで「毎回守る約束事」を固定化できます。 ([OpenAI Developers][2])
（Codex CLI でも /init で AGENTS.md を作る導線があります。） ([OpenAI Developers][3])

1) 「Issue＝Codexプロンプト」設計の基本フォーマット

各Issueは、Codexに渡すプロンプトとして以下の構造で統一するのが最も安定します。

Issue構造の基本要素

  • Objective：このIssueで達成すること（1〜3行）
  • Scope：対象ファイル・追加するモジュール・やらないこと
  • Requirements：仕様（型、I/O、例外、ログ、再現性）
  • Acceptance Criteria：完了条件（チェックボックスで）
  • Verification：Codexが実行するコマンド（pytest 等）
  • Notes：実装上の注意（依存追加禁止、破壊的変更禁止…）

この形式が「小さく分ける」「検証可能にする」というCodex推奨に合致します。 ([OpenAI Developers][1])

2) まずリポジトリに置く「Codexの永続コンテキスト」：AGENTS.md

Codexは作業前に AGENTS.md を読みます。ここに"全Issue共通の規約"を書いておくと、Issue本文を短くできます。 ([OpenAI Developers][2])

提案：リポジトリ直下 AGENTS.md（コピペ用）

``` 
# AGENTS.md (CIBG v0.1)

## Working agreements (must follow)
- Use Python 3.11+.
- Prefer small, reviewable diffs.
- Do NOT add new runtime dependencies unless the issue explicitly allows it.
- Always run tests after changes:
  - python -m pip install -e ".[dev]"
  - pytest -q
- Keep the public API stable unless the issue explicitly requests changes.
- Use type hints for public functions and dataclasses/pydantic models.
- Keep generation deterministic: honor "seed" whenever applicable.
- Write or update tests for any new behavior.

## Project layout
- Package code in: src/cibg/
- Tests in: tests/
- Export artifacts (json/midi) into out_dir/ produced by CLI.

## Style
- Prefer clear names and short functions.
- Avoid cleverness. Make behavior easy to test.
```

3) GitHub Issueテンプレ（Codex入力フォーム）提案

テンプレを .github/ISSUE_TEMPLATE/codex_task.md としてコミットしておくと運用が楽です。

.github/ISSUE_TEMPLATE/codex_task.md

```

## Context
- Related docs/spec: docs/spec.md (if exists)
- Depends on: #<issue>

## Scope
**In scope**
- ...

**Out of scope**
- ...

## Requirements
- ...

## Acceptance Criteria
- [ ] Code compiles and imports
- [ ] Tests added/updated
- [ ] `pytest -q` passes
- [ ] CLI / output works if relevant

## Verification (Codex must run)
```bash

```

* Keep diffs minimal.

```

4) CIBG v0.1 のIssue分解（Milestone設計）

ここから下は、そのままIssueとしてコピペできる本文です。
Issue番号（#1など）はあなたのGitHub側で振られるので、本文中は「Depends on:（前提Issue）」を タイトルで参照する形にしています（番号は後で置換でOK）。

M [META] Issue（全体管理）

Title

[META] CIBG v0.1 実装バックログ（Codex Issue運用）

Body（コピペ用）

``` 
## Objective
CIBG v0.1 を段階的に実装する（Issue＝Codexプロンプトとして運用）。

## Milestone scope
- Input型: BeatCondition (pydantic)
- Processing: Planner + DenoiseLoop (operators/score/anneal)
- Output: BeatArtifact (spec/pattern/json + MIDI)

## Issue order (recommended)
1. chore: repo bootstrap + AGENTS.md + CI + docs/spec.md
2. feat(types): core data models (condition/spec/pattern/artifact)
3. feat(io): json I/O + artifact exporter
4. feat(planner): rule-based planner
5. feat(model): noise initializer
6. feat(model): score model v0
7. feat(model): operator framework + meter_fix
8. feat(model): snare_anchor + hat_density
9. feat(model): scheduler + annealer
10. feat(model): denoise loop integration
11. feat(render): MIDI renderer
12. feat(cli): `cibg generate` end-to-end
13. feat(model): extra operators (kick/bass/variation)
14. docs: README usage + examples

## Definition of Done (v0.1)
- condition.json -> out_dir に spec.json/pattern.json/drums.mid/bass.mid が出る
- seed固定で再現可能
- pytest が通る
```

5) 実装Issue（Codexプロンプト）一式

以下のIssue本文は そのままGitHub Issue本文に貼って使えます。

1 リポジトリ基盤

Title: chore: repo bootstrap (pyproject/src layout/pytest/CI) + AGENTS.md + docs/spec.md

Labels: codex chore P0

最初の土台を作成。プロジェクトスケルトン、テスト設定、CIワークフローを実装。

2 Input/Outputの型

Title: feat(types): implement core models (BeatCondition/BeatSpec/Pattern/BeatArtifact)

Labels: codex feature P0

pydantic v2モデルを実装。JSON検証/シリアライゼーションを提供。

3 JSON I/O

Title: feat(io): JSON load/save + export artifact folder structure

Labels: codex feature P0

JSONロード/セーブ機能と標準化されたout_dirフォルダ構造を実装。

4 Planner

Title: feat(planner): rule-based planner (BeatCondition -> BeatSpec)

Labels: codex feature P0

決定論的ルールベースのプランナーを実装。BeatConditionからBeatSpecを生成。

5 白色ノイズ初期化

Title: feat(model): noise initializer (init_white_noise_pattern)

Labels: codex feature P0

ノイズパターン初期化を実装。シードに基づく決定論的生成を提供。

6 ScoreModel v0

Title: feat(model): score model v0 (constraints + density + anchor + variation)

Labels: codex feature P0

パターン品質評価のためのスコアモデルを実装。制約、密度、アンカー、バリエーションを考慮。

7 Operator基盤

Title: feat(model): operator framework + O_meter_fix

Labels: codex feature P0

Operatorフレームワークと最初のオペレータ（O_meter_fix）を実装。

8 snare_anchor + hat_density

Title: feat(model): O_snare_anchor + O_hat_density_shape

Labels: codex feature P0

スネアアンカーとハット密度シェイプの2つのオペレータを実装。

9 ConditionScheduler + Annealer

Title: feat(model): condition scheduler (phase) + annealer accept/reject

Labels: codex feature P0

フェーズマッピング、温度スケジュール、受理ルールを実装。

10 DenoiseLoop（統合コア）

Title: feat(model): denoise loop integrates initializer + operators + score + anneal + debug

Labels: codex feature P0

コア生成ループを実装。ノイズパターンから反復的な改良を実行。

11 MIDIレンダラ

Title: feat(render): MIDI renderer (drums/bass) with swing + hat_roll

Labels: codex feature P0

PatternからMIDIファイルへのレンダリングを実装。スウィングタイミングとハットロールをサポート。

12 CLI cibg generate

Title: feat(cli): implement cibg generate end-to-end (condition.json -> out_dir artifacts)

Labels: codex feature P0

エンドツーエンドのCLIコマンドを実装。condition.jsonからアーティファクトを生成。

13 追加オペレータ

Title: feat(model): add operators O_kick_anchor / O_bass_follow / O_variation_between_bars

Labels: codex feature P1

音楽性を向上させる3つの追加オペレータを実装。キックアンカー、ベースフォロー、バリエーション。

14 ドキュメント整備

Title: docs: README usage + pipeline overview + example output description

Labels: codex docs P1

ドキュメントを改善。新規コントリビュータがCIBG v0.1を迅速に実行できるように。

注意：Issue本文の完全な内容

上記は各Issueの概要のみです。完全なIssue本文（Objective, Scope, Requirements, Acceptance Criteria, Verification, Notes）は元のドキュメントに含まれており、そのままGitHub Issueにコピーできます。

6) 運用メモ（GitHub×Codexで回すときのコツ）

  • 1 Issue = 1 PRの粒度にする（今回の分解はそうなっています）
  • Issue本文には必ず Verificationコマンドを書く（Codexが自己検証できる）
  • 共通ルールは AGENTS.md に集約（Issue本文の重複を削れる）
必要なら、上のIssue群を「GitHub Projects / Milestone」向けに、P0/P1の優先度・依存関係（DAG）として並べ替えた版も作れます（ただ、いま提示した順番がそのまま実装順として最短です）。
