型駆動Python学習計画

論文モデル(G:(W,E)→(S,A(S)))を学習設計に転用

🎯 1日2時間 × 3ヶ月（約90日 = 約180時間）のPythonトレーニング計画

✨ 型選択レンズで「プログラミング＝型を選ぶこと」を体得

0. 設計モデル（添付論文）を学習設計に当てはめる

論文の定義：

\[ G : (W,E)\rightarrow (S,A(S)) \]
  • (W)：連続的な可能性の集合（"世界"）
  • (E)：制約・評価基準
  • (S)：観点（何を区別したいか）＝観測関数 \(o_S\)
  • (A(S))：観点 (S) における離散的な同値類（等価クラス集合）

これを学習に翻訳すると：

  • (W)：Pythonで書けるあらゆるプログラム／課題／設計の可能性（膨大で連続）
  • (E)：学習時間・学習ゴール・難易度・道具・あなたの教えたい骨格（＝「型を選ぶ」）
  • (S)：学習者に"区別して選べるようになってほしい型の観点"
  • (A(S))：その観点で区別される「型の選択パターン（型レシピ）」の集合 → この (A(S)) をカバーするように90日メニューを配置します

1. (W) と (E) の定義（この計画の入力）

(W)（Type-driven Python Programming World）

あなたの骨格に合わせて、世界 (W) を次のように見ます：

プログラミング＝「入力の型」×「モデル（変換器）の型」×「出力の型」を選び、必要なら副作用・例外も型として扱うこと

ここでいう「型」は Python の int/str/list だけではなく、広い意味での型です：

  • データ型：int, float, str, list[T], dict[K,V] …
  • モデル（変換器）の型：
    • 純粋関数 Tin -> Tout
    • 状態を持つオブジェクト（クラス）
    • ジェネレータ（遅延列）
    • コンテキストマネージャ（資源の型）
    • 例外／Result型（失敗の型）
    • 非同期（async）など
  • 副作用（Effect）：I/O、ネットワーク、更新（ミューテーション）も"型っぽく"扱う

(E)（制限・評価基準）

  • 時間：1日2時間×約3ヶ月（約90日）
  • 学習優先：「型を選ぶ」発想で書けること（暗記より設計）
  • 出力物：ミニ課題（週1）、中間プロジェクト（月1）、最終プロジェクト（最後の2週間）
  • 技術制約（おすすめ）：標準ライブラリ中心＋必要最小限
    • typing, dataclasses, pathlib, json, csv, unittest/pytest（好みで）
    • 外部ライブラリは「最後に味見」程度にしてもOK

2. 中間生成物：(S)（観点）を明示する

論文では観点 (S) は 観測関数 (\(o_S:W\to O_S\)) でした。
この学習計画では、あなたの骨格をそのまま観測関数にします。

観点 (S)：型選択レンズ（Type Selection Lens）

各課題 (\(w\in W\)) に対して、次を観測できるようにします：

\[ o_S(w)=( T_{\text{in}}, T_{\text{model}}, T_{\text{out}}, T_{\text{effect}}, T_{\text{error}} ) \]
  • (\(T_{\text{in}}\))：入力の型（例：str, list[str], dict[str, int], Path…）
  • (\(T_{\text{model}}\))：モデル（変換器）の型 例：純粋関数 / ジェネレータ / クラス / コンテキストマネージャ / async など
  • (\(T_{\text{out}}\))：出力の型（例：int, list[int], Report など）
  • (\(T_{\text{effect}}\))：副作用の型（none / file I/O / network / mutation）
  • (\(T_{\text{error}}\))：失敗の型（例外で落とす / Optional / Result風データ など）

学習目標は「Python文法を知る」ではなく
この5つを"自分で選び直せる"ようになることです。

3. 中間生成物：(A(S))（等価クラス集合）＝「型レシピ」一覧

(A(S)) は「観点 (S) で同じに見えるものの集まり＝等価クラス」でした。
学習では、等価クラスを "頻出の型選択パターン（型レシピ）"として離散化します。

ここでは コア24個に絞ります（90日で回しやすい粒度）。

(A(S))：型レシピ（コア24）

A1〜A6：スカラー／文字列（純粋関数中心）

  1. A1 int -> int（計算、丸め、範囲）
  2. A2 str -> int（パース、失敗戦略）
  3. A3 str -> float（小数・指数表記・エラー）
  4. A4 str -> str（正規化：strip/lower/replace）
  5. A5 str -> bool（バリデーション predicate）
  6. A6 Any -> str（整形・表示・フォーマット）

A7〜A12：コレクション変換（map/filter/reduce の型）

  1. A7 list[T] -> list[T]（filter）
  2. A8 list[T] -> list[U]（map）
  3. A9 Iterable[T] -> Iterator[U]（generator pipeline）
  4. A10 Iterable[T] -> U（集計：sum/min/max/len）
  5. A11 dict[K,V] -> V（lookup + default）
  6. A12 Iterable[T] -> dict[K, list[T]]（grouping）

A13〜A16：失敗・境界・例外（"失敗も型"）

  1. A13 Optional[T] -> T（None処理：デフォルト/例外）
  2. A14 Tin -> Tout | Exception（例外設計）
  3. A15 Tin -> Result[Tout, Error]（失敗を値として返す）
  4. A16 Tin -> Tout + 境界値の分割（空、0、最大、異常…）
    ※論文の「EP→BVA（精密化）」に対応

A17〜A20：I/O境界（Effect を型として扱う）

  1. A17 Path -> str/bytes（読み取り）
  2. A18 str/bytes -> Path（書き込み）
  3. A19 str <-> dict（JSON serialize/parse）
  4. A20 Iterable[dict] <-> CSV（構造⇄テキスト）

A21〜A24：モデル型（関数→クラス→状態機械）

  1. A21 dict -> dataclass（スキーマ化）
  2. A22 dataclass -> dict/str（出力モデル）
  3. A23 State + Event -> State（状態遷移：クラス/関数）
  4. A24 Config -> Service（依存性：設定から構築）

90日メニューは「毎日なにをやるか」ですが、
設計としての本体はこの (A(S))（型レシピ集合）をどう被覆するかです。
＝論文の言い方だと「等価クラス集合を被覆する代表元（課題）を選ぶ」。

4. 1日2時間×3ヶ月の「メニュー」（90日想定）

1日の基本フォーマット（固定）

時間 内容
0:00–0:15 今日の型レンズ：Tin / Tmodel / Tout / Effect / Error を先に決める
0:15–0:45 あなたがライブ実装（短い完成品）
0:45–1:35 お友達が演習（少なくとも2本）
1:35–2:00 振り返り：
  • 今日の型レシピ（A番号）
  • 失敗パターン（境界・例外・None）
  • テスト観点（EP/BVA）をメモ

7. この設計が論文モデルに沿っている点（要約）

  • あなたが教えたい骨格を 観点 (S)（観測関数）として定義した → Tin/Tmodel/Tout/Effect/Error
  • その観点で学習世界 (W) を離散化したものが (A(S))（型レシピ24）
  • 90日メニューは、(A(S)) を被覆する代表課題の選択として設計した （論文でいう「等価クラスを被覆するテスト集合選択」に対応）
  • さらに毎週、EP→BVA（精密化）を必ずやらせて「型を選ぶ」筋肉を鍛える

必要なら、あなたが考えている「最終プロジェクトの題材」（例：家計簿、ログ解析、在庫、スクレイピングではなくローカル解析等）を1つ仮決めして、上の Week12〜Day90 をその題材に合わせて Tin/Toutの具体型まで落とした版に"即"作り替えられます。
