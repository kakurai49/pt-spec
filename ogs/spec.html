OGS (Obligation Gate System)

Spec / 技術スタックと設計指針

FastAPI・Redis・Celeryを組み合わせて、PULL運用と監査ログを両立させる実装設計。 「状態」と「監査」を分離し、GateBoardとEvaluationが正確に収束することを優先します。

Scope Index Overview Spec
技術スタックの役割

FastAPI

キャンペーン作成、queue一覧、claim、結果提出、evaluationなどの主要APIを提供。

Redis

状態管理・インデックス・監査イベント（Streams）を保持し、読み取り用途と監査用途を分離。

Celery

Ready再計算、放置claim回収、滞留検知などの非同期処理を担当。

主要データモデル
データ役割
CampaignWorldSnapshotと対象obligationを束ねる単位。
ObligationCataloga1〜a12のDoD・証跡・依存関係を定義。
ObligationItemキャンペーンごとの実行単位。Queueに積まれる。
Evidence実施した証跡（ログ/CI/リンクなど）。
AuditEventclaim/submit/evidence/exception/judgeを記録。
ExceptionApproved例外承認の理由・承認者・時刻を保持。
GateBoarda1〜a12の進捗と合格可否を集計表示。
主要APIの整理
API目的備考
POST /campaignsキャンペーン作成WorldSnapshotと期間を登録
POST /campaigns/{id}/initobligation初期化ObligationItem生成とQueue投入
GET /queueREADY一覧WIP制限を反映
POST /queue/{item}/claimclaim二重claim防止・冪等性
POST /items/{id}/submit結果提出PASS/FAIL/Blocked
POST /items/{id}/evidenceEvidence追加監査イベント同時記録
POST /items/{id}/exception例外承認理由・承認者・時刻
GET /gateboardGateBoardExit Criteriaに基づく集計
POST /judge-okJudge OK最終判断を監査ログへ
Celeryタスク（非同期処理）
  • Ready再計算：DoD達成度と依存関係をもとにQueueを更新。
  • 放置claim回収：期限超過のclaimを解放し再投入。
  • 滞留検知：Blockedや長期停滞を検出して通知。
タスクは冪等性を前提にし、再実行に耐える設計とする。
技術スタッフ向けチェックリスト
  1. API操作が監査イベントとして必ず記録されるか。
  2. claim/submit/evidenceの状態遷移が矛盾しないか。
  3. GateBoard/Evaluationが最新状態を反映できるか。
  4. Celeryタスクが冪等で再実行可能か。
  5. 例外承認（ExceptionApproved）が監査可能か。
