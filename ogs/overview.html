OGS (Obligation Gate System)

Overview / 学習&応用マニュアル

PULL型運用とobligation中心の品質合格モデルを、初心者エンジニアが理解・応用できるように整理した学習用マニュアル。 Doneの曖昧さを排除し、Evidenceと監査ログで判断できる運用を作ります。

Scope Index Overview Spec
システムの特徴（要約）
  • PULL型運用：仕事は「割当」ではなく「引く」。WIP制限と二重claim防止を設計。
  • obligation中心：テスト技法ではなくa1〜a12の合格義務で品質を収束。
  • Evidence / 監査ログ：実施した証跡がJudge OKを支える。
  • 例外承認フロー：ExceptionApprovedも監査可能な状態で記録。
  • GateBoard可視化：a1〜a12の進捗と品質合格を即座に把握。
主要コンセプト（W / E / S* / A(S*)）
記号意味OGSでの扱い
W連続的な可能性空間キャンペーンごとにWorldSnapshotを持つ
E制約／評価情報E_quality（Exit Criteria）とE_pull（PULL制約）
S*ゲートobligation観点テスト技法ではなくobligation視点で整理
A(S*)キューに積める離散単位ObligationItemとしてa1〜a12を生成
観点を「技法」ではなく「合格義務」に置き換えることで、運用と判断が明確になります。
役割と責務

member

Queueからclaimし、実行結果とEvidenceを提出する。

leader

キャンペーン作成・obligation初期化・Judge OKを担う。

admin

システム全体の管理権限。例外承認の最終責任者。

reviewer

Evidenceのレビュー承認を担当（任意）。

典型フロー（初心者の道順）
  1. キャンペーン作成（leader/admin）
  2. Obligationカタログ確認（全ロール）
  3. obligation初期化（leader/admin）
  4. キュー確認 & claim（member/leader/admin）
  5. 結果提出（PASS/FAIL/Blocked）
  6. Evidence追加
  7. レビュー承認（任意）
  8. GateBoard / Evaluation確認
  9. Judge OK確定
機能別の学習ポイント

Obligation Catalog

DoD・証跡・依存関係をデータとして定義し、運用と実装を分離する。

Queue / Claim

READYな項目のみを表示し、WIP制限と二重claim防止を仕組みにする。

結果提出 & Blocked

DoDを満たした後にPASS/FAILを確定し、Blockedは理由と解決タスクを残す。

Evidence

監査に耐える証跡構造を用意し、評価判断の裏付けに使う。

ExceptionApproved

例外でも必ず理由・承認者・時刻を記録し、GateBoardで注記付きでPass扱い。

GateBoard / Evaluation

a1〜a12の進捗と品質合格可否をリアルタイムに可視化する。

監査イベント（Audit Events）

主要操作はAudit Eventsとして記録し、トラブルシュートと監査対応に備える。

  • claim / submit / evidence / exception / judge を必須イベントとして残す。
  • イベントと状態更新の順序（イベント先行 or 状態先行）を明示。
  • GateBoardは状態データ、監査はイベントデータを参照。
