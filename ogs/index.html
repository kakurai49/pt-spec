OGS (Obligation Gate System)

Index / 進め方

PULL型運用とobligation運用を10日で一気に把握するためのガイド。 1日のアウトプットは「GateBoardに映る状態」と「Evidenceが残る状態」です。

Scope Index Overview Spec
標準フロー（学習の順番）
  1. キャンペーン作成：目的と期間、対象ロールを決める。
  2. Obligationカタログ確認：a1〜a12のDoD・証跡要件を読み解く。
  3. obligation初期化：キャンペーンごとにObligationItemを生成しQueueへ。
  4. claim → 実行：memberがPULLでclaimし、WIP制限を守って実行する。
  5. 結果提出 + Evidence：PASS/FAIL/Blockedと証跡を同時に残す。
  6. レビュー承認：必要に応じてreviewer/adminが承認する。
  7. GateBoard確認：a1〜a12の達成状況を可視化して確認。
  8. Judge OK：leader/adminが最終判断を監査ログに記録。
各ステップは「実施した証跡が残ること」を合否判断の中心に置きます。
10日プラン例
Dayテーマアウトプット
1OGS全体像W/E/S*/A(S*)を自分の言葉で説明
2Obligation Cataloga1〜a12のDoDと証跡要件を整理
3Queue / ClaimWIP制限・二重claim防止のルール設計

5ExceptionApproved例外承認フローと記録項目を整備
6GateBoard可視化に必要な状態データを設計
7EvaluationGreen/Yellow/Red判定条件を定義
8Judge OK最終判断の監査ログを設計
9技術スタックFastAPI/Redis/Celeryの役割を整理
10総まとめGateBoardとAuditの整合性チェック
提出物の粒度
  • 必ず「どのobligation（a1〜a12）に影響したか」を明記する。
  • Evidenceの種類（ログ/CI/リンク/画像など）を列挙する。
  • Blockedの場合は理由と解決タスクをPULL化する。
Doneの曖昧さを排除するため、Evidenceの欠落は合格扱いにしません。
